Anvyll · Anvyll — UX study

Sessions & context, without the detective work

Your tester wasn't confused because the feature is bad — the session/context system is genuinely strong. They were confused because the overlay hides all of its state and vocabulary behind one ⋮ menu. This is the redesign: make context visible, give it one home, and let the words explain the model.

01 · Where a first-time user gets lost

Five frictions, straight from the current UI

Each of these is observable in Overlay.tsx today. Together they explain the “little bit confusing” verdict.

Context state is invisible

The only hint that a session or project is active is the placeholder text in the empty output field. Once anything is typed or generated, there is zero indication of which session you're in, whether a project is attached, or whether context will be used at all.

“Is it… using my context right now? How do I know?”

Everything lives behind the ⋮ menu

+ New session, Clear session, the session list, and Configure context all sit inside one overflow menu — split across two side-by-side panels. Overflow menus are for rare actions; here they hide the product's second-biggest idea.

Six words, no visible hierarchy

Session, Project, Context, Import, Configure, Scope. The UI never shows that a project contains sessions — sessions are a flat list with colored dots, and projects only appear in a separate picker. The user must reverse-engineer the data model from dot colors.

The import round-trip explains how, never why

“Copy this prompt into a chat with your other AI provider” is a clever flow, but the modal never says what the payoff is (your refinements will understand your work). The Session|Project scope pill sits at the bottom, easy to miss — paste into the wrong scope and nothing warns you.

Deep nesting for a simple intent

“Start fresh on my project” = ⋮ menu → + New session → “Start with project context?” picker → maybe + New Project → import modal (scope locked) → back to picker. Four glass surfaces for one intent, and no feedback loop afterwards showing what the context actually did.

02 · The model users need to hold

Three layers of context — say so, out loud

The engine already stacks three kinds of grounding (contextLayer.ts, session.ts). The fix is not new architecture — it's naming the layers in the UI in the user's own words.

Automatic

Where you're typing

App, window, selection, file names — captured on the hotkey, zero effort. Today this layer is completely silent.

Show it: “Detected: Cursor · main.ts” — one read-only line, builds trust for free.

Session

What you're working on right now

The current goal — imported once per conversation, dies with it. Internally: SessionContext, titled from its GOAL line.

Speak it: “This session” — and always show its title on the card, not just in a menu.

Project

What's always true about your project

Stack, conventions, names — standing knowledge that survives across sessions. Internally: ProjectContext with an accent color.

Speak it: “Project memory” — a folder that contains sessions, shown as a real hierarchy.

03 · Design principles

Four rules for the redesign

A — State is visible

Context lives on the card, not in a menu

A one-line context bar at the top of the overlay always shows project dot · session title. Click it → the one context panel. No more guessing.

B — One home

Merge four menu items into one panel

New session, resume, clear, import, project switch — one popover, one hierarchy: projects as groups, sessions nested inside. ⋮ keeps only Position / Settings / Studio / Dismiss.

C — Words teach the model

Rename to the user's vocabulary

“Configure context” → Context. “Import context” → Bring context from your chat. Empty states explain the payoff in one sentence before asking for any work.

D — Close the loop

Show what grounded each refinement

After Refine, a quiet chip row: Grounded by: Session · Project · Cursor. Context stops being a leap of faith — the user sees it working.
